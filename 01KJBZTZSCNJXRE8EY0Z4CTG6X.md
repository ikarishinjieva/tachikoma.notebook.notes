# 经验

  - "只进行分析, 列出几种可能, 不要修改代码"
  - 维护一个需求描述文件, 并且每次参考这个文件, 使用这个文件中的需求项进行检查, 防止需求丢失
    - 更进一步: 维护一个code_analysis方案, 其中包括 界面-API-主要功能流程-主要代码函数 的对应关系
      - 样例: [code-analyze.txt](/assets/01KJBZTZSCNJXRE8EY0Z4CTG6X/code-analyze.txt)
  - 对于出现的问题, 先分析, 然后打印日志, 不要进行修复
  - 用唯一的数据结构, 维持界面上所有的数据, 让界面变成状态无关, 这样AI最容易识别及维护
  - 代码迁移: 使用claude高级模型, 现制定计划, 然后要求按照计划执行. 
    - 如果用其他模型直接执行, 会出现迁移过程中修改逻辑的问题, 或者遗漏步骤  
  

  - 进行功能设计, 不要修改代码 -> 进行简化 -> 进行代码规划, 不要修改代码 -> 实现代码, 不要测试
  - 将易错的组件功能, 比如面包屑的功能细节, 写一个组件说明

  - 模拟模式, 从AI函数模拟, 别让大模型自定模拟端 (已犯错)
