# 主要思路

SARGAM 将代码生成拆分成: 召回代码样例 -> 生成初步代码 -> 进行细粒度修改, 这三个步骤

其主要创新是, 在"细粒度修改"这一步骤中, 使用LevT之类的 (针对 代码修改 进行训练的) 模型, 对初步代码进行修订

相比使用大型语言模型 (LLM) 直接进行代码修订，使用 LevT 类似的模型，即**基于编辑操作的模型**，主要有以下优劣势：

**优势：**

1\. **更高的编辑效率:** LevT 类模型专注于预测细粒度的编辑操作（插入、删除、替换），而不需要像 LLM 那样从头生成整个代码序列。由于代码修改通常只是对局部进行调整，因此 LevT 类模型可以更高效地完成任务。  
2\. **更精准的修改:** 由于直接预测编辑操作，LevT 类模型可以更精准地控制修改过程，避免 LLM 可能产生的语法错误或逻辑错误。  
3\. **更易于解释:** LevT 类模型的输出是具体的编辑操作序列，更易于开发者理解模型的修改意图，方便进行人工检查和干预。  
4\. **训练数据利用率更高:** 相较于需要大量文本数据进行预训练的 LLM，LevT 类模型更专注于代码编辑任务，可以使用更少的训练数据达到较好的效果。

**劣势：**

1\. **泛化能力可能不足:** LevT 类模型的优势在于对局部修改的精准控制，但对于需要较大范围代码理解和重构的任务，其泛化能力可能不如 LLM。  
2\. **需要更精细的预处理:** 为了让 LevT 类模型能够正确预测编辑操作，需要对输入的代码进行更精细的预处理，例如标记编辑位置、识别代码结构等。  
3\. **模型设计和训练更复杂:** 相比于直接使用预训练 LLM 进行微调，设计和训练一个有效的 LevT 类模型需要更多领域知识和工程经验。

**总结:**

* 对于**局部代码修改、语法纠错、代码风格统一**等任务，LevT 类模型通常是更优的选择，因为它们效率更高、修改更精准、更易于解释。  
* 对于**需要代码理解、逻辑推理、代码生成**等更复杂的任务，LLM 则更有优势，因为它们具备更强的语义理解和生成能力。

在实际应用中，可以根据具体需求选择合适的模型，甚至可以将 LLM 和 LevT 类模型结合起来，例如使用 LLM 生成初步的代码修改建议，再使用 LevT 类模型进行细化和优化。
