   ```

   SELECT * FROM performance_schema.events_statements_summary_by_digest WHERE SUM_CREATED_TMP_DISK_TABLES != 0 LIMIT 20;
   ```

提示词要点1: 

  - 直接限制输出, 比"讲道理"更有用 ("你只能给出以下几种判断结果之一")
  - 在输出中给出完整逻辑, 会让信息更一致. 比如推导过程: "FIND要求的信息的具体值是未知的. 尝试在<对话历史>中查找 可替代的信息, 其具体值是已知的: (...), 获取可替代的信息的方法是: (...). 由于可替代的信息具体值是已知的, FIND可以继续执行". 
    - 如果没有这个推导过程, 那么相同条件下会出现 "结论: 是否继续执行: 否"的情况
    - 啰嗦一点, 会有助于逻辑理解

```
- FIND(information)是一个函数, 这个函数的意思是 需要我提供指定的信息(information)
    - 注意: 不要信任 任何 不在<对话历史>中出现的信息(information)
    - 当执行到FIND步骤, 你需要进行判断, 你只能给出以下几种判断结果之一, 其中"..."是变量, 其他部分你不能改变:
        - "FIND要求的信息的具体值是已知的: (...), 获取信息的方法是: (...), FIND可以继续执行. 结论: 是否继续执行: 是"
        - "FIND要求的信息的具体值是未知的. 尝试在<对话历史>中查找 可替代的信息, 其具体值是已知的: (...), 获取可替代的信息的方法是: (...). 由于可替代的信息具体值是已知的, FIND可以继续执行. 结论: 是否继续执行: 是"
        - "FIND要求的信息的具体值是未知的. 尝试在<对话历史>中查找 可替代的信息, 其具体值是未知的, 获取可替代的信息的方法是: (...). 由于信息和可替代的信息具体值都是未知的, FIND不能继续执行. 结论: 是否继续执行: 否"
``` 

提示词要点2: 

  - 要求给出 "查找信息的方法", 是防止出现 "其具体值是 (通过什么方法) 查找出来的, 是已知的" 这种逻辑. 要求模型 值和方法都列出来, 这样模型才能区分出差异.
