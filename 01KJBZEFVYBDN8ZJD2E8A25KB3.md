# 主要内容

仿照设计模式, 论文提出了提示词的设计模式: 

```
论文中提到的16种提示模式目录以及简要的解释如下：

**1. 输入语义 (Input Semantics)**

* **元语言创建 (Meta Language Creation)**:  允许用户创建自定义语言或符号系统，并向LLM解释其语义，以便在后续交互中使用更简洁、清晰的表达方式。

**2. 输出定制 (Output Customization)**

* **输出自动化器 (Output Automater)**:  当LLM的输出建议用户执行一系列步骤时，该模式会指示LLM生成可自动执行这些步骤的脚本或其他自动化工具。
* **角色扮演 (Persona)**:  为LLM设定一个角色或身份，例如安全专家、Linux终端等，使其从特定角度生成输出，并关注特定类型的细节。
* **可视化生成器 (Visualization Generator)**:  指示LLM生成可被其他可视化工具（如Graphviz Dot、DALL-E）理解的文本输出，从而将文本信息转化为图表、图像等更直观的表现形式。
* **步骤清单 (Recipe)**:  用户提供部分已知的步骤或“原料”，LLM生成一个完整的步骤序列，以实现用户设定的目标，并识别出不必要的步骤。
* **模板 (Template)**:  用户预先定义一个输出模板，LLM根据模板的结构和占位符生成内容，确保输出格式符合特定需求。

**3. 错误识别 (Error Identiﬁcation)**

* **事实核查清单 (Fact Check List)**:  指示LLM生成一个包含其输出所依赖的关键事实的清单，以便用户核实这些事实的准确性，从而评估输出的可靠性。
* **反思 (Reﬂection)**:  要求LLM对自己的输出进行反思，解释其推理过程、依据和假设，帮助用户理解LLM的决策过程，并发现潜在的错误或偏差。

**4. 提示改进 (Prompt Improvement)**

* **问题优化 (Question Reﬁnement)**:  指示LLM分析用户的问题，并根据领域知识和上下文信息，提出更准确、更完善的版本，引导用户进行更有效的提问。
* **备选方案 (Alternative Approaches)**:  当用户提出一个任务或问题时，LLM不仅提供解决方案，还会列出其他可行的备选方案，并比较它们的优缺点，帮助用户做出更全面的决策。
* **认知验证器 (Cognitive Veriﬁer)**:  指示LLM将用户的问题分解成一系列更具体、更容易回答的子问题，并根据用户对子问题的回答，生成更准确的最终答案。
* **拒绝突破器 (Refusal Breaker)**:  当LLM拒绝回答问题时，该模式会指示LLM解释拒绝的原因，并提供可行的替代问题或措辞建议，帮助用户绕过限制，获得所需信息。

**5. 交互 (Interaction)**

* **反转交互 (Flipped Interaction)**:  将对话的控制权交给LLM，让LLM主动向用户提问，以收集必要的信息，完成特定任务，例如进行测验或生成部署脚本。
* **游戏化 (Game Play)**:  指示LLM围绕特定主题创建一个游戏，LLM负责生成游戏内容、规则和交互流程，用户可以通过游戏的方式学习新知识或技能，例如论文中提到的网络安全游戏。
* **无限生成 (Inﬁnite Generation)**:  指示LLM根据预先定义的提示和规则，持续生成一系列输出，直到用户明确指示停止，例如无限生成人物姓名和职业的URL。

**6. 上下文控制 (Context Control)**

* **上下文管理器 (Context Manager)**:  允许用户明确指定LLM在生成输出时应考虑或忽略哪些上下文信息，例如特定主题、代码规范等，以提高输出的相关性和准确性。
```
