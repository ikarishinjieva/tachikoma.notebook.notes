# 论文目标

提出REAPER，一个基于推理的规划器，使用一个较小的语言模型（LLM）生成高效的检索计划。

检索计划, 类似于CoT的思考过程, 决定应该从哪里检索什么关键字, 调用哪个Agent

# 目标问题

REAPER的核心在于其提示设计和用于生成多样化训练数据的模块。

REAPER使用了三个模块来解决训练数据的问题

# DQS (Diverse Query Sampler)模块

* 目的： 减少模型偏差，增强其对不同用户查询的泛化能力。  
* 方法： 自动生成语义上不同的用户查询样本，确保训练数据能够覆盖各种不同的用户需求和表达方式。

步骤要点: 

  - 去除最近和最远的极端样本
  - 使用与标注样本相同数量的非标注样本 (但后续仍需要人工标注)

我尝试根据论文的描述，更详细地解释一下 DQS 的工作步骤，并补充一些例子：

**DQS 的目标是：**  从一个较大的用户查询池中，筛选出一个语义上多样化的子集，用于训练 REAPER。

**DQS 的工作步骤如下：**

1. **准备两个查询集合：**
    *  `Qinitial`： 一个高质量的、人工标注的初始查询集合，包含了各种典型的用户查询。
    *  `Qlarge`： 一个更大的、未经标注的查询集合，可以是通过人工收集、规则生成或从真实用户日志中采样得到的。

2. **计算查询嵌入 (Query Embedding):**  使用预训练的语言模型 (例如 RoBERTa) 分别计算 `Qinitial` 和 `Qlarge` 中每个查询的向量表示。

3. **计算语义相似度：**  计算 `Qinitial` 中每个查询与 `Qlarge` 中所有查询之间的余弦相似度，得到一个相似度矩阵 `S`。

4. **识别极端样本：**  根据相似度矩阵 `S`，找到 `Qlarge` 中与 `Qinitial` 中查询最相似和最不相似的样本，构成极端样本集合 `Qextreme`。

5. **移除极端样本：**  从 `Qlarge` 中移除 `Qextreme` 中的样本，得到精简后的查询集合 `Qrefined = Qlarge \ Qextreme`。

6. **随机采样：**  从 `Qrefined` 中随机采样与 `Qinitial` 数量相同的查询，构成最终的多样化查询集合 `Qdiverse`。

**例子：**

假设我们有以下查询集合：

* `Qinitial` =  {"这款手机的电池续航时间怎么样？", "这款手机支持无线充电吗？"}
* `Qlarge` = {"这款手机的相机怎么样？", "手机的价格是多少？", "电池能用多久？", "这款手机有指纹解锁吗？", "支持无线充电功能吗？", "手机的防水性能如何？"}

经过步骤 2 和 3 计算语义相似度后，我们假设得到以下结果 (为了简化例子，这里只展示了部分相似度)：

| 查询 | "电池能用多久？" | "支持无线充电功能吗？" |
| --- | --- | --- |
| "这款手机的电池续航时间怎么样？" | 0.8 | 0.3 |
| "这款手机支持无线充电吗？" | 0.2 | 0.9 |

可以看出，"电池能用多久？" 与 "这款手机的电池续航时间怎么样？" 语义比较相似，而 "支持无线充电功能吗？" 与 "这款手机支持无线充电吗？" 语义非常相似，因此这两个查询可能会被归类为极端样本 (`Qextreme`)。

最终，DQS 模块会从剩余的查询中随机选择两个，构成多样化的查询集合 `Qdiverse`。

# **TEvo (Tool Evolve) 模块**

  - 目的： 增强模型对工具描述的理解能力，使其不会过度依赖特定的工具名称或描述方式。
  - 方法： 对工具提示进行语义上的改写，例如使用同义词替换工具名称，用不同的方式描述工具功能，或者调整示例代码的格式等，同时保持最终生成的检索计划不变。

(REAPER会输出 调用哪些工具, 所以对工具的描述需要进行多样化调整)

# TTG (Tool-Task Generator) 模块

  - 目的： 提高模型对检索计划的理解能力，使其能够处理各种不同的任务需求。
  - 方法： 将主要任务（生成检索计划）转换为多个相关任务，例如：
    - 从计划生成查询：给定一个检索计划，让模型推断出用户最初的查询是什么。
    - 完成部分计划：给定一个不完整的检索计划，让模型补充缺失的步骤。
    - 识别正确的工具：给定一个用户查询，让模型判断应该使用哪些工具来获取信息。
    - 掩码步骤完成：给定一个部分被掩盖的检索计划，让模型预测被掩盖的步骤。
    - 重新排序步骤：给定一个乱序的检索计划，让模型将其重新排列成正确的顺序。
    - 掩码参数识别：给定一个部分参数被掩盖的检索计划，让模型预测被掩盖的参数。
    - 限制计划中的工具：给定一个用户查询和一个受限的工具集，让模型只使用这些工具生成检索计划。

主要思路: 将查询转换成不同角度的QA, 增强LLM的任务理解和泛化能力
