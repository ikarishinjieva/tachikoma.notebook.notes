# 主要创新点

将RL引入Text2SQL

  1. "冷启动": 先sft, 再RL
  2. 定制了奖励函数: 
     1. SQL-R1 使用了一个 **复合的、渐进式的反馈机制** 作为强化学习的奖励函数。这个奖励函数由四种类型的奖励组成，旨在从不同层面评估和引导模型生成的 SQL 候选和推理过程：

        - **格式奖励 (Format Reward, Sf)：**

          - **目的：** 鼓励模型输出符合预设的格式规范。
          - **评估内容：** 检查推理过程是否包含在 `...` 标签内，最终 SQL 答案是否包含在 `...` 标签内，并且 SQL 语句本身是否包含在 `sql...` 标签内。
          - **奖励值：** 格式正确得 +1 分，格式不正确得 -1 分。
        - **执行奖励 (Execution Reward, Se)：**

          - **目的：** 评估生成的 SQL 候选的语法正确性，防止模型生成无法执行的无效 SQL。
          - **评估内容：** 尝试执行生成的 SQL 候选，并限制执行时间。
          - **奖励值：** 如果 SQL 候选可执行得 +2 分；如果格式不正确（则不进行执行检查）得 0 分；如果 SQL 候选不可执行得 -2 分。**如果格式不正确，后续的奖励（结果奖励和长度奖励）将不会计算。**
        - **结果奖励 (Result Reward, Sr)：**

          - **目的：** 评估执行 SQL 候选后得到的查询结果是否与真实的、正确的查询结果一致，这是衡量 SQL 准确性的重要标准，也是奖励机制中的关键部分。
          - **评估内容：** 比较执行生成的 SQL 候选得到的结果与标准答案的结果（使用执行准确率 EX 进行评估）。
          - **奖励值：** 如果查询结果正确得 +3 分；如果格式不正确或 SQL 候选不可执行得 0 分；如果查询结果不正确得 -3 分。
        - **长度奖励 (Length Reward, Sl)：**

          - **目的：** 激励模型生成更全面和详细的推理过程（即 `` 标签内的内容）。
          - **评估内容：** 这个奖励只在 **查询结果正确** 的情况下计算。它由两部分构成：
            - 一部分基于总回答长度（思考过程长度 + 答案长度）与设定的最大回答长度的比例。
            - 另一部分基于 SQL 语句长度占整个答案长度（`` 标签内的内容长度）的比例，这旨在避免答案中包含过多不必要的解释。
          - **奖励值：**
            - 如果查询结果正确且总回答长度小于等于最大长度：`0.5 × Stl + Sal`
            - 如果查询结果正确且总回答长度大于最大长度：`0.5 + Sal` （对超长进行部分惩罚）
            - 其他情况（结果不正确等）：`0`  
其中，`Stl = (思考过程长度 + 答案长度) / 最大长度`，`Sal = SQL 长度 / 答案长度`。
