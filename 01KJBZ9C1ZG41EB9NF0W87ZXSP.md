# 主要思路

  - 通过RAG进行代码召回, 召回 <需求, 代码> 对
  - 将 <需求, 代码> 对, 整理成 <需求, 中间结果, 代码> 对. 
    - 中间结果, 指的是这个 代码 的一些结果的表达, 比如 对应的测试用例, 或者对应的API. 能方便LLM理解这个代码的效果
  - 将整理完的结果, 给LLM, 让LLM对新需求进行编码

举例: 

```
假设用户需求是“写一个函数，判断一个三角形是否是等腰三角形”。

1. **示例检索：** AceCoder 首先在代码库中检索相似需求的代码，例如“判断一个三角形是否是直角三角形”，“计算三角形的面积”等。
2. **提示构建：** AceCoder 将检索到的代码转换成三元组示例，例如：
    * 需求：“判断一个三角形是否是直角三角形”
    * 初步结果：测试用例：输入三个边长，判断是否满足勾股定理。
    * 代码：实现判断直角三角形的代码。
3. **代码生成：** AceCoder 将构建好的提示输入 LLM，LLM 首先根据提示生成判断等腰三角形的测试用例，然后根据测试用例和示例代码生成判断等腰三角形的代码。
``` 

# 有用的信息

  - 生成中间结果, 能辅助LLM理解代码用途
  - 对于召回的代码, 论文中提到了一种去重算法 (基于词匹配), 将重复的结果去除
