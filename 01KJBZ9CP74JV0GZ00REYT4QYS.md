==========

背景：

客户反应mysql报错：“the row size is 8148 which is greater than maximum allowed size (8126) for a record on index leaf page.”

[图片1]

[附件1: ci表结构.txt]

分析：
分析报错发现在对表tt_ci添加列的时候报错，跟客户要了一下表tt_ci的表结构；

发现tt_ci表字段比较多，根据客户描述，对表增加字段的操作是业务sql自动增加的，目前客户没有返回具体的业务sql。

客户要求：
通过修改MySQL使业务不在报错

给客户建议：
通过在本地环境进行场景复现，验证关闭参数innodb_strict_mode可以让业务不在报错，但是会在mysql错误日志中出现告警信息。目前可以从DMP上关闭这个参数。

参数含义：
这个参数设置会影响CREATE TABLE、ALTER TABLE、CREATE INDEX和OPTIMIZE TABLE语句的语法错误处理；innodb_strict_mode还支持记录大小检查，这样INSERT或UPDATE操作就不会因为记录太大而失败。

当innodb_strict_mode被禁用时，InnoDB会忽略冲突子句，在创建表或索引时只在消息日志中显示一个警告。生成的表可能具有与预期不同的特征，例如在尝试创建压缩表时缺乏压缩支持。

重新跟客户沟通：
进一步研究这个参数，最终跟客户表述了如果关闭这个参数对mysql的影响，建议从业务sql下手进行排查，尽量让业务sql满足对应版本的mysql的相关要求。客户已跟业务进行同步，目前没有反馈是否开启innodb_strict_mode参数。

详细处理和验证参考宗杨泰隆银行工单：[链接1]

对主从复制影响：
关闭innodb_strict_mode参数，主库业务可以正常进行，过了两天发现从库延迟较大，复制中断，检查从库innodb_strict_mode也是关闭状态，但是从库报错“the row size is 8148 which is greater than maximum allowed size (8126) for a record on index leaf page.”

[图片2]

[图片3]

经过验证发现从库的sql线程还是处于修改innodb_strict_mode参数之前的状态，这个参数如果修改，只会对修改之后生成的连接生效，对之前存在的连接不生效，所以会导致复制中断。

在从库执行start slave可以重新恢复复制正常。

原因：
参数innodb_strict_mode是一个session和global级别参数，在dmp上对参数进行修改时，只会对主库和从库的global级别进行调整，在这之前的线程不能被马上生效，导致从库sql线程还是处于开启innodb_strict_mode状态，但是在从库上发现innodb_strict_mode参数是关闭的，容易引发误导，所以一定要看参数是否既是session又是global级别。

==========
