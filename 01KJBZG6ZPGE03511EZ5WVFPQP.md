# 论文目的

在Text to SQL过程中, 引入两种辅助纠错工具:

  - 检索器 (Database Retriever): 主要发现 条件子句 中的常量值 与 实际值 不匹配的情况
  - 检测器 (Error Detector): 检测更多的逻辑错误, 比如: 
    - **执行错误:** SQL语法错误、数据库schema错误（例如引用不存在的表或列）。
    - **更严格的约束不匹配:**
      - 外键关系不匹配
      - JOIN操作冗余或缺失
      - 条件子句中列类型不匹配
      - GROUP BY子句缺失或使用不当
      - 用户自定义的约束违反 (例如特定列的空值处理、特定数据格式要求等)

# 两种纠错工具的步骤

```
让我们分别详细描述检索器和检测器的步骤：

**检索器 (Database Retriever)**

1. **触发条件:** 当LLM生成的SQL语句中的条件子句（WHERE或HAVING子句）在数据库中找不到匹配的条目时，触发检索器。

2. **输入信息:** 检索器接收以下输入：
    * 数据库 `D`
    * 用户问题 `Q`
    * LLM生成的SQL语句中的条件子句 (例如 `WHERE T2.d = X`)

3. **检索过程:** 使用SimCSE模型，将条件子句中出现的值（例如 `X`）与数据库 `D` 中对应列（例如 `T2.d`）的所有单元格进行语义相似度计算。

4. **输出结果:** 返回与值 `X` 语义最相似的几个数据库单元格，例如 `['v1', 'v2', 'v3', 'v4', 'v5']`。  这些相似单元格作为反馈提供给LLM代理。

5. **LLM代理的后续操作:**  LLM代理根据检索器返回的相似单元格，进行以下操作：
    * 如果LLM认为原始条件正确，则选择其中一个最接近的单元格值（ground truth cell）替换原始值 `X`，例如将 `WHERE T2.d = X` 修改为 `WHERE T2.d = 'v1'`。
    * 如果LLM认为原始条件错误，则根据反馈重新生成一个新的条件子句。
    * 为了避免重复操作，每次LLM生成新的条件子句后，都会从上下文中的数据库schema中移除之前用过的条件列。

**检测器 (Error Detector)**

1. **输入信息:** 检测器接收以下输入：
    * 数据库 `D`
    * 用户问题 `Q`
    * LLM生成的SQL语句对应的函数调用序列 `Y` (例如 `add_answer(T2.a)`, `add_join(T1.b, T2.c)`, `add_where(T2.d, =, X)` 等)

2. **提取数据库Schema信息:**  从数据库 `D` 中提取所有表名、列名、列类型、外键关系等schema信息。

3. **错误检测程序:**  检测器内部有一系列预定义的错误检测程序，用于检查函数调用序列 `Y` 中的参数是否符合SQL语法规则和数据库schema。

4. **检测的错误类型:** 检测器主要检测以下类型的错误：
    * **执行错误:** SQL语法错误、数据库schema错误（例如引用不存在的表或列）。
    * **更严格的约束不匹配:**
        * 外键关系不匹配
        * JOIN操作冗余或缺失
        * 条件子句中列类型不匹配
        * GROUP BY子句缺失或使用不当
        * 用户自定义的约束违反 (例如特定列的空值处理、特定数据格式要求等)

5. **检测过程:**  检测器遍历函数调用序列 `Y` 中的每个函数调用，并根据其参数和数据库schema信息，运行相应的错误检测程序。

6. **输出结果:**  如果检测到错误，则返回具体的错误信息 `ξ` 给LLM代理，例如 "No such foreign key relationship: T1.b=T2.c" 或 "No such column: T2.a"。

7. **LLM代理的后续操作:** LLM代理根据检测器返回的错误信息，修改SQL语句对应的函数调用序列，并重新生成SQL语句。

总而言之，检索器专注于**条件子句的值的正确性**，而检测器专注于**SQL语句的语法和结构的正确性以及更广泛的约束满足情况**。它们共同帮助LLM代理生成更准确、更有效的SQL查询。
``` 

# 使用LLM的函数调用

使用LLM的函数功能来生成SQL, 能增强SQL的结构化正确性.

举例: 

````
让我们用例子来说明针对LLM的函数调用设计和处理子查询的QA()函数的作用。

**假设有如下数据库表:**

* `products` 表:  `id`, `name`, `category`, `price`
* `categories` 表: `id`, `name`

**用户问题:** 找出价格高于所有电子产品平均价格的玩具产品的名称。

**1. 针对LLM的函数调用设计:**

传统的Text-to-SQL方法需要LLM直接生成完整的SQL语句，例如：

```sql
SELECT T1.name
FROM products AS T1
WHERE T1.category = (SELECT id FROM categories WHERE name = 'Toys')
  AND T1.price > (SELECT AVG(price) FROM products WHERE category = (SELECT id FROM categories WHERE name = 'Electronics'));
```

而使用论文中提出的函数调用设计，LLM只需要生成以下函数调用序列：

```python
add_answer(products.name)
add_from(products)
add_where(products.category, =, QA('玩具的类别ID是什么？'))
add_where(products.price, >, QA('电子产品平均价格是多少？'))
```

其中，`QA()`函数用于处理子查询。

**2. 处理子查询的QA()函数:**

对于上面的例子，LLM需要处理两个子查询：

* **子查询1：玩具的类别ID是什么？**  LLM会为这个子查询单独生成函数调用序列：

```python
add_answer(categories.id)
add_from(categories)
add_where(categories.name, =, 'Toys')
```

* **子查询2：电子产品平均价格是多少？** LLM会为这个子查询单独生成函数调用序列：

```python
add_answer(AVG(products.price))
add_from(products)
add_where(products.category, =, QA('电子产品的类别ID是什么？'))
```

这里又出现了一个嵌套的子查询：

* **子查询3：电子产品的类别ID是什么？** LLM会为这个子查询单独生成函数调用序列：

```python
add_answer(categories.id)
add_from(categories)
add_where(categories.name, =, 'Electronics')
```

**最终结果:**

通过将SQL语句分解成一系列函数调用，并使用`QA()`函数处理子查询，LLM可以逐步生成最终的SQL语句。  框架会将这些函数调用最终组装成完整的SQL语句，并交给数据库执行。

**优势:**

* 简化了LLM的任务，使其更容易生成结构正确的SQL。
* 可以方便地处理嵌套的子查询。
* 提高了LLM生成复杂SQL语句的能力。

这个例子展示了函数调用设计和`QA()`函数如何协同工作，帮助LLM生成包含子查询的复杂SQL语句。  这种方法使得LLM可以专注于理解用户问题的语义，而无需直接处理复杂的SQL语法。
```` 

# Spider-Mismatch数据集

这个数据集 它着重强调用户问题中提到的值与数据库实际值之间的差异，这种差异在真实场景中非常常见，但现有数据集很少涉及。 这要求模型不仅仅能理解问题的语义，还要能够有效地处理这种值的不匹配，例如大小写不一致、缩写、同义词等。

其他特点: 

```
Spider-Mismatch 数据集是从 Spider、Spider-SYN 和 Spider-DK 数据集中衍生出来的，专门用于突出真实世界场景中条件不匹配的问题，并以此挑战 Text-to-SQL 模型。其主要特点如下：

1. **关注条件不匹配：**  这是 Spider-Mismatch 的核心特点。它着重强调用户问题中提到的值与数据库实际值之间的差异，这种差异在真实场景中非常常见，但现有数据集很少涉及。  这要求模型不仅仅能理解问题的语义，还要能够有效地处理这种值的不匹配，例如大小写不一致、缩写、同义词等。

2. **基于现有数据集改造：** Spider-Mismatch 并非完全从头构建，而是从 Spider、Spider-SYN 和 Spider-DK 的验证集中筛选和修改而来。  这保证了数据集的质量和多样性，同时也使其更具可比性。

3. **手动引入干扰：**  为了模拟真实场景中的条件不匹配，数据集的创建者手动对问题和对应的 SQL 查询进行了修改，引入了多种干扰，例如：
    * **移除列名：**  在问题中移除明确的列名提及，使问题更口语化，但也增加了模型理解问题的难度。
    * **移除高亮标记：**  移除问题中对值的特殊标记（例如引号、首字母大写等），使问题更自然，但也增加了模型识别值的难度。
    * **替换常见值：**  将问题中的一些常见值替换为其他值，例如将日期、数字等替换为其他日期、数字，这要求模型能够根据上下文理解值的含义，而不是简单地进行字符串匹配。

4. **包含字符串类型单元格：**  数据集的构建者特意选择了包含字符串类型单元格的例子进行修改。这是因为字符串类型的数据更容易出现不匹配的情况，例如大小写不一致、缩写、同义词等。

5. **去除重复和简单示例：**  为了避免数据冗余和降低任务难度，数据集的构建者去除了重复的例子和一些过于简单的例子。

6. **更具挑战性：**  由于引入了各种干扰和更贴近真实场景的条件不匹配，Spider-Mismatch 比现有的数据集更具挑战性，能够更好地评估 Text-to-SQL 模型在实际应用中的性能。

通过以上这些特点，Spider-Mismatch 数据集弥补了现有数据集的不足，为 Text-to-SQL 研究提供了一个更贴近真实场景、更具挑战性的基准，推动了该领域朝着更实用化的方向发展。
```
