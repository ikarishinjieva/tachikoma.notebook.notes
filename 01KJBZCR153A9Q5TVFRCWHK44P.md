# 论文分析了 "LLM+KG" 范式和"LLM * KG" 范式的区别

# 

```
论文中对 "LLM+KG" 范式和 "LLM * KG" 范式都进行了介绍，但详细程度有所不同：

**1. "LLM*KG" 范式:**

* 论文中称其为 "routine"，也就是现有研究中常用的方式。
*  **主要思路:**  从KG中检索信息，根据检索到的信息增强prompt，然后将增强的prompt输入LLM。
*  **局限性:**  LLM在推理过程中扮演的角色更像是一个 "translator"，将自然语言问题转化为机器可理解的查询语句，并没有直接参与到图推理过程中。这种松耦合的方式使得LLM的推理能力受限，并且高度依赖于KG的完整性和质量。
*  **论文中的例子:** 图1b 展示了这种范式在KG信息缺失的情况下如何导致推理失败。

**2. "LLM * KG" 范式:**

* 论文重点介绍的**新范式**。
*  **主要思路:**  KG和LLM协同工作，在图推理的**每个步骤**中相互补充，形成更紧密的耦合关系。
*  **优势:**  
    *  **深度推理:**  ToG从KG中提取多样化和多跳的推理路径，增强了LLM的深度推理能力。
    *  **可靠推理:**  明确可编辑的推理路径提高了LLM推理过程的可解释性，并支持对模型输出来源的追踪和修正。
    *  **灵活性:**  ToG是一个即插即用的框架，可以无缝地应用于各种LLM和KG。
    *  **高效性:**  ToG可以利用小型LLM实现与大型LLM相当的推理能力，降低了部署和应用成本。
*  **论文中的例子:**  图1c 展示了这种范式如何通过LLM的动态推理能力和KG中检索到的知识来克服 "LLM+KG" 范式的局限性。

总而言之，论文对 "LLM+KG" 范式进行了简要介绍，并将其作为对照组，重点突出了 "LLM * KG" 范式的创新性和优势。论文中提出的ToG算法框架就是 "LLM * KG"  范式的一种具体实现。
```

# 算法的主要步骤

```
论文中描述的 "Think-on-Graph" (ToG) 算法框架包含三个主要阶段：

**1. 初始化 (Initialization):**

   *  **目标:**  确定推理路径的起始实体，初始化 top-N 推理路径集合  `P`。
   *  **步骤:**
      1.  利用 LLM 自动提取问题中的主题实体。
      2.  将得分最高的 top-N 个主题实体作为推理路径的起始节点，构建初始的 `P`。

**2. 探索 (Exploration):**

   *  **目标:**  迭代地扩展推理路径，寻找与问题相关的知识。
   *  **步骤:**  在每一轮迭代 (depth `D`) 中：
      1.  **关系探索 (Relation Exploration):**
         *   **搜索 (Search):**  对于 `P` 中的每条路径，查询 KG 获取尾实体的所有邻接关系，构成候选关系集合 `RD_cand`。
         *   **剪枝 (Prune):**  利用 LLM 根据问题文本和 `RD_cand` 中的关系，选择出 top-N 个最相关的关系，更新 `P`，并将选出的关系构成新的尾关系集合 `RD`。
      2.  **实体探索 (Entity Exploration):**
         *   **搜索 (Search):**  对于 `P` 中的每条路径，根据尾实体和 `RD` 中对应的关系，查询 KG 获取相应的头实体或尾实体，构成候选实体集合 `ED_cand`。
         *   **剪枝 (Prune):**  利用 LLM 根据问题文本和 `ED_cand` 中的实体，选择出 top-N 个最相关的实体，更新 `P`，并将选出的实体构成新的尾实体集合 `ED`。

**3. 推理 (Reasoning):**

   *  **目标:**  判断当前推理路径是否足以回答问题，并生成最终答案。
   *  **步骤:**
      1.  利用 LLM 判断当前 `P` 中的推理路径是否包含足够的信息来回答问题。
      2.  如果判断结果为“是”，则利用 LLM 整合 `P` 中的信息生成最终答案。
      3.  如果判断结果为“否”，且未达到预设的最大搜索深度，则返回步骤 2 继续探索；否则，返回无法回答。

**简要概括:**

ToG 算法框架的核心思想是利用 LLM 引导在 KG 上进行beam search，迭代地探索和构建推理路径。在每一轮迭代中，LLM 都参与到关系和实体的选择过程中，确保推理路径与问题高度相关。最终，LLM 根据推理路径生成最终答案。
``` 

beam search, 类似于用cost剪枝的进行广度优先.

ToG算法, 相当于给LLM提供信息, 让LLM决定搜索路径

# 论文还提出了ToG-R算法

```
你的理解基本正确，可以再精确一些：

* **ToG (Think-on-Graph):**  LLM 并非只根据 KG 的边和点来决定如何搜索，而是在每一步的搜索过程中，**结合问题文本和 KG 中的信息**，对候选的边和点进行评估和选择。可以理解为 LLM 在 KG 上进行"带着问题"的探索，而非盲目搜索。

* **ToG-R (Think-on-Graph with Relation):**  LLM 的作用更侧重于 **预测关系**，而不是仅仅评估 KG 中已有的关系。它会根据问题文本和当前推理路径，预测下一步最有可能连接到答案的关系，然后利用 KG 来验证这个预测是否正确，即是否存在对应的路径。

**简要对比：**

| 特征 | ToG | ToG-R |
| --- | --- | --- |
| 推理粒度 | 三元组 (边-点-边) | 关系链 (边-边-边...) |
| LLM 作用 | 评估和选择 KG 中的边和点 | 预测关系，并利用 KG 验证预测结果 |
| KG 作用 | 提供推理路径的候选项 | 验证 LLM 预测的关系是否存在对应路径 |

**举例说明：**

假设我们要回答问题："谁是苹果公司的 CEO？"

* **ToG:**  LLM 会从"苹果公司"这个实体出发，探索 KG 中所有与之相连的边和点，例如"创始人"、"产品"、"CEO"等，并根据问题文本判断哪条路径更有可能找到答案。
* **ToG-R:**  LLM 会直接预测与问题相关的关键关系，例如"CEO"，然后在 KG 中验证是否存在连接"苹果公司"和某个实体的"CEO"关系。

总而言之，ToG-R 相比 ToG 更加"目标导向"，它利用 LLM 的预测能力来引导 KG 的搜索，从而提高了推理效率。
``` 

相当于: 

ToG算法, 只根据图谱的信息, 规划搜索路径

ToG-R算法, 使用信息对关系进行预测, 用图谱进行验证, 这样速度更快但更不精确
