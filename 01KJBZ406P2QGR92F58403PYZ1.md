# 综述

ConflictBrokerV1 是最朴素的冲突检测, 对于记录的唯一键进行冲突检查和排队

ConflictBrokerV2 带有热点表的事务合并

背景: [20231117 - 在Oracle并发压力中, 测试OMS性能 [1]], 场景2, 性能问题是由于ConflictBrokerV1对热点表的事务进行了冲突排序, 并发度过低

# 相关参数

  - hotKeyMerge: 是否启用 ConflictBrokerV2
  - hotKeyTable: json格式 "[{schema: xxx, tableName: xxx}]", 定义hotKeyMerge机制涉及的表
  - mergeDelayThreshold: 冲突事务延迟超过此阈值时, 才会触发合并
  - conflictPrintSize: 冲突事务超过此阈值, 才会触发合并. 默认值256

# 逻辑

事务进入冲突检测器: ConflictBrokerV2 与 ConflictBrokerV1 相同: 

  - 对于事务中的每一行数据: 
    - 抽取数据的唯一键值
    - 在冲突表中查询与其冲突的前置事务, 将当前事务和前置事务设置依赖关系
    - 如果不存在冲突的前置事务, 则将当前事务输出, 送入sink进行回放

事务回放完成后, 从冲突检测器中清理的逻辑 (release), ConflictBrokerV2 与 ConflictBrokerV1有区别: 

  - 对于(已经完成回放的)事务中的每一行数据
    - 对于每一个唯一键值
      - 获取 其冲突列表 (其他事务的数据 集合)
      - 判断是否可合并 (事务级别), 需要同时满足以下条件: (com.oceanbase.oms.connector.dispatcher.scheduler.ConflictBrokerV2#canMerge)
        - 冲突列表长度 >= conflictPrintSize
        - 该事务的延迟 (距离当前时间) >= mergeDelayThreshold
      - 开始合并 (对于 db.table.col 进行合并)
        - 当已经合并的事务数小于128时, 可以继续合并
        - 遍历冲突列表, 比较 前一数据和当前数据 是否可合并 (数据级别), 需要满足以下条件:
          - 对于 (两个事务链) 合并 的链结构检查
          - 记录变更类型 的兼容性 (recordTypeMatch)
            - 比如: DELETE和INSERT合并, 等
          - 唯一键不能变更, 且两条数据记录的唯一键相同
        - 合并方法: 将前置事务中的数据的值改成最终事务的值, 这样几个事务就可以并发
