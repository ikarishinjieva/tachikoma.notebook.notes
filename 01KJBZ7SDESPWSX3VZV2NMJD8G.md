# 让AI复述, 能增强AI的理解

比如

```
你需要按照如下步骤进行思考, 并将思考过程输出 (思考过程可以参考问题诊断路径):
<思考步骤>
0. 告诉我当前要解决的问题是什么

1. 在问题诊断路径中, 我们处于哪个节点
 
2. 如果<我的输入>是一个信息(而不是个问题), 你需要结合问题内容和对话历史对 这个信息是否有助于解决问题 进行评论, 并简短描述原因
- 在评论中，需要说明输入是否和问题相关
- 在评论中，需要对输入内容进行简要描述
- 在评论中，需要说明输入的内容是否可能导致问题出现
 
3. 结合问题内容、对话历史、以及对输入的评价：
- 列出一个或者两个，你认为最相关，最能帮助确认问题原因的关键信息，并需要具体说明理由
- 当出现和问题强相关的输入时，比如是关键信息中的一种，你应该即刻识别它作为解决问题的关键信息
 
 
4. 一个问题的解决被分成了四个阶段: 问题诊断阶段 -> 问题解决方案评估阶段 -> 问题解决方案实施阶段 -> 后续评估与预防阶段.
- 当输入信息有如下特点的时候，你需要将问题诊断阶段标记为已完成，并进入问题解决方案评估阶段
       - 已经能够识别可能的原因
 
5. 结合问题内容、对话历史、以及对输入的分析判断，告诉我当前的问题已经到了什么阶段
 
7. 根据目前的信息, 如果对于问题的解决可以总结出阶段性的结论, 你需要简短描述这个阶段性结论
 
8. 根据目前的信息, 告诉我在解决问题的过程中, 你还需要哪些信息 (需要齐全地进行列举):
8.1. 列出问题诊断阶段还需要哪些信息
8.2. 列出问题解决方案评估阶段还需要哪些信息
8.3. 列出问题解决方案实施阶段还需要哪些信息
8.4. 列出后续评估与预防阶段还需要哪些信息
 
9. 按照已经获得的信息, 加上问题诊断阶段需要获取的信息, 列出目前存在几种可能的解决方案
</思考步骤>
``` 

前两个问题: "在问题诊断路径中, 我们处于哪个节点", "告诉我当前要解决的问题是什么" 都是让AI对信息进行复述. 

如果不进行复述, 那么之后的思考步骤中的问题对 这两部分的信息会出现误判 (比如: 认为当前要解决的问题是另一个问题)

复述能增强理解

# 让AI进行分类, 能增强理解能力

仍如上例, 对信息进行了分类: 

```
8.1. 列出问题诊断阶段还需要哪些信息
8.2. 列出问题解决方案评估阶段还需要哪些信息
8.3. 列出问题解决方案实施阶段还需要哪些信息
8.4. 列出后续评估与预防阶段还需要哪些信息
``` 

如果不这样处理, 而直接让AI列出 "问题诊断是需要什么信息", 那么很多其他信息仍会被列出.

在分类情况下, 这些"其他信息"会被列在其他阶段, 不会列入问题诊断

# "虚拟和现实" 是个良好的分界符

仍如上例, 使用了"以上参考资料中, 是虚构的场景和解决思路, 不是本次的真实场景" 作为参考资料和问题背景的分界

使用其他分界词, 会导致AI将参考资料中的一些问题描述, 作为实际发生的问题的背景进行分析, 导致错误. 

"虚拟和现实"这种分界符, 对通义千问很适用

# 使用代码(进行模拟执行), 更容易让AI理解逻辑

对比使用mermaid格式来梳理逻辑: 

````
根据参考资料, 建议的问题诊断路径如下 (mermaid格式): 
```
[graph TD

subgraph 排查INSERT语句执行慢
A[检查慢查询日志] --> B[分析慢查询日志中的SQL语句]
B --> |存在锁等待| C[查看当前锁情况]
C --> |锁等待时间长| D[查看innodb_lock_wait_timeout参数设置]
C --> |锁等待时间正常| E[查看innodb_buffer_pool_size参数设置]
B --> |不存在锁等待| F[查看表结构和索引]
F --> |缺少索引| G[添加索引]
F --> |索引存在| H[优化SQL语句]
end

]
```
```` 

使用代码格式来梳理逻辑, AI能更好的理解和执行逻辑: 

```
if 能获取慢查询日志:
    检查慢查询日志
    if 存在慢查询:
        分析慢查询日志中的SQL语句
        if 存在锁等待:
            查看当前锁情况
            if 锁等待时间长:
                查看innodb_lock_wait_timeout参数设置
            if 锁等待时间正常:
                查看innodb_buffer_pool_size参数设置
        if 不存在锁等待:
            查看表结构和索引
            if 缺少索引:
                添加索引
            if 索引存在:
                优化SQL语句
if 不能获取慢查询日志:
    检查processlist中的SQL
``` 

可以让AI进行执行: 

```
<问题诊断代码>的输入是<人类的输入>, 你需要判断: <问题诊断代码>能执行到哪一步, 下一步的代码是什么, 为什么不能继续执行, 还缺什么信息 
    
    执行代码时, 需要满足以下要求: 
    - 从头开始执行, 不要跳过任何步骤
    - 如果某些步骤不能完成, 则停在该步骤

    你需要回答"可以执行到X步骤, 接下来可以进行Y步骤, 不能继续执行的原因是: ..."
``` 

但使用mermaid格式执行, AI会跳过步骤, 不能逐步执行

# 增加主语, 能让结果更稳定

```
"如果没有明确提供信息xxx, 不能完成执行"
 
这种输入中, 一定概率会被误解成 AI没有提供信息, 输出会是: "信息是用什么命令能获取的, 能完成执行", 其中信息命令是AI内置的知识
 
改成"如果人类没有明确提供信息xxx, 不能完成执行", 会稳定的避开上述错误
``` 

# 明确的逻辑梳理, 比 指明方向 效果更好

以下的强制思考模板 (应该尽量使用条件分支和嵌套的逻辑判断，使得排查步骤结构化，保证不同的情况采取不同的分析路径) 比 通用思考模板 (每一步骤和分析应清晰解释其逻辑和必要性，确保整个排查过程的逻辑链条清晰无误) 的效果更好

```
<强制思考模板>
- 不能出现任何<参考文档>中原文的直接引用，尤其是对表名、字段名、文件名的引用或者截取等操作
- 排查计划中不应该有任何知识假设，每个基础步骤都应该在PLAN中出现
- MySQL的故障，通常需要数据库的操作、数据库配置，日志分析等多个方面，因此在优化PLAN时不能有关注偏差
- 排查计划应从原始问题的最初现象开始分析，聚焦于实际现象中存在的异常（如报错信息），而不是从一般的可能性分析开始
- 排查计划在保证流程的逻辑性和系统性的基础上，还应该直接关联问题的重要环节，对特定问题进行深入挖掘
- 应该尽量使用条件分支和嵌套的逻辑判断，使得排查步骤结构化，保证不同的情况采取不同的分析路径
- FIND函数、ANALYSE函数和IF函数，通常具有先后逻辑关系（先采集素材，然后分析素材，最后对素材异常进行判断）
- 排查计划以THINKING(), FIND(), IF()为单位，对问题相关的异常和异常原因进行确认和验证；直到最终执行REASON_CONFIRMED()和RETURN()
- 在一个CASE中，第一轮的THINKING()的内容为第二轮THINKING()的内容的前提，第二轮THINKING()的内容为第一轮的THINKING()的内容的扩展
- 必须避免在一个CASE中，THINKING()内容相同或者毫无关系的情况
</强制思考模板>
 
 
<通用思考模板>
- 思考: 排查步骤提供了详细的指令和方法，如检查错误日志、系统日志、硬件状态等，这有助于系统地进行故障诊断
- 思考: 对人类提供的信息进行评价
- 思考: 步骤按照逻辑顺序排列，从基础的日志检查到具体的函数分析和源码审查，有助于逐步缩小问题范围
- 思考: 提供了具体的命令或操作指南，确保排查能够针对性地执行
- 思考: 排查步骤按照逻辑顺序组织，从基本检查到深入分析，有助于系统地识别和解决问题
- 思考: 步骤间的过渡自然，便于追踪问题发展脉络
- 思考：明确指出了具体的问题区域或功能模块，使得问题的定位更为精确
- 思考：通过关联具体的系统指标或错误信息，提高了诊断的针对性和有效性
- 思考：使用实际的数据或日志输出来支持分析，增强了结论的可靠性
- 思考：基于数据进行决策，确保分析结果具有客观性和实际可操作性
- 思考：包含从表面到深层的多个检查点，确保全面覆盖可能的故障源
- 思考：通过逐步深入的方法，逐层剥离可能的问题，直至找到根本原因
- 思考：在执行每一步后，应包含对结果的验证，确保每一步骤的正确性和有效性
- 思考：缺乏验证可能导致后续步骤建立在错误的基础上，影响整体排查的准确性
- 思考: 每一步骤和分析应清晰解释其逻辑和必要性，确保整个排查过程的逻辑链条清晰无误
- 思考：在排查步骤中应考虑实施效率，避免资源密集型或时间消耗大的步骤，特别是在紧急情况下
- 思考：当引用相关的bugs或历史问题时，应详细说明它们与当前问题的直接关联
- 思考：提供充分的证据来支持任何假设或结论，确保其具有足够的说服力
- ...
</通用思考模板>
```

# 简化原理

在学习"动态规划"时, 论文会列出复杂的公式.

这是可以要求:

  - 给出一个动态规划的经典的案例: 0-1背包
  - 在这个案例上, 询问公式哪些变量是固定的
  - 要求模型将某一个变量 放在 0-1背包的故事中, 说明其可变时, 故事哪里变了
