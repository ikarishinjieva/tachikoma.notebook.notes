# 论文目标

RAGCHECKER是在claim (断言) 级别, 评估 检索器和生成器 的质量的框架

# RAGCHECKER细节

```
RAGCHECKER 评估的是 RAG 系统生成回复的质量，但它评估的具体内容比简单的质量好坏要更细致，它评估的是回复在 **claim 级别** 的质量以及产生这个质量的 **原因**，并分别从 **整体、检索器和生成器** 三个层面进行评估。

具体来说，RAGCHECKER 评估以下内容：

1. **Claim 级别的正确性:**  RAGCHECKER 将回复和参考答案分解成一个个 claim（断言），然后判断回复中的 claim 是否正确。 这比单纯比较回复和参考答案的相似度更精细，因为它关注的是回复中每个具体信息的准确性。

2. **Claim 级别的完整性:**  RAGCHECKER 评估回复是否包含了参考答案中所有的 claim。这反映了回复的全面性和信息覆盖程度。

3. **检索器的有效性:**  RAGCHECKER 评估检索器能否找到与问题相关的文本块。它关注两个方面：
    * **Claim Recall (CR):**  参考答案中的 claim 有多少被检索到的文本块所覆盖。
    * **Context Precision (CP):** 检索到的文本块中有多少是与参考答案相关的。

4. **生成器的能力:** RAGCHECKER 评估生成器如何利用检索到的信息生成回复。它关注以下几个方面：
    * **Faithfulness (Faith.):**  回复中的 claim 有多少能在检索到的文本块中找到依据。这反映了生成器是否忠实于检索到的信息。
    * **Relevant Noise Sensitivity (NS(I)):**  回复中错误的 claim 有多少来自相关的文本块。这反映了生成器对相关文本块中噪声的敏感性。
    * **Irrelevant Noise Sensitivity (NS(II)):** 回复中错误的 claim 有多少来自不相关的文本块。这反映了生成器对不相关文本块中噪声的敏感性。
    * **Hallucination (Hallu.):**  回复中错误的 claim 有多少在检索到的文本块中找不到依据。这反映了生成器“幻觉”的程度。
    * **Self-Knowledge (SK):**  回复中正确的 claim 有多少在检索到的文本块中找不到依据。这反映了生成器依赖自身知识的程度。
    * **Context Utilization (CU):**  检索到的文本块中与参考答案相关的 claim 有多少被用在了回复中。这反映了生成器对检索到的信息的利用率。

5. **整体性能:**  RAGCHECKER 也评估 RAG 系统的整体性能，使用 Precision、Recall 和 F1 值来衡量回复的质量。

通过这些细粒度的评估，RAGCHECKER 不仅能判断 RAG 系统生成回复的质量，还能帮助开发者找出问题所在，例如是检索器的问题还是生成器的问题，从而更有针对性地改进系统。
``` 

论文中还提及了一些根据评估结果进行改进的建议: 

```
论文基于对8个不同RAG系统在10个领域数据集上的实验结果以及RAGCHECKER指标的分析，提出了一系列改进RAG系统的建议，主要集中在**检索器**和**生成器**两个方面：

**检索器方面：**

* **改进检索模型：** 实验结果表明，使用更强大的检索模型（例如 E5-Mistral 相比于 BM25）可以显著提升整体性能。 因此，建议优先考虑使用效果更好的嵌入模型和检索算法。

* **适度增加上下文数量 (k)：** 增加检索的文本块数量可以提高 Claim Recall，为生成器提供更多信息，从而提升 Faithfulness。 但需要注意的是，过多的上下文也会引入更多噪声，降低 Context Precision，并增加生成器的 Noise Sensitivity。 因此，需要根据实际情况选择合适的 k 值，并权衡信息量和噪声之间的关系。

* **适度增加文本块大小：**  增加文本块大小的效果与增加 k 值类似，可以提供更多上下文信息，但也会增加噪声。  在有限的上下文长度限制下，建议使用较大的文本块和较小的 k 值，尤其是在比较简单的任务上。  这可以提高 Context Precision，并减少噪声的影响。

* **文本块重叠率的影响较小：**  实验发现调整文本块的重叠率对整体性能的影响不大。  虽然更高的重叠率可以略微提高 Context Precision，但并不会显著增加检索到的有用信息总量。  因此，在实际应用中，可以不必过多地调整这个参数。

**生成器方面：**

* **在 Prompt 中加入明确的要求：**  实验表明，在生成 prompt 中明确要求生成器提高 Faithfulness、Context Utilization 并降低 Noise Sensitivity，可以有效引导生成器朝着这些方向优化。  例如，可以要求生成器“必须基于提供的文档进行回答”，“不要包含文档中没有的信息”等。

* **平衡 Context Utilization 和 Noise Sensitivity：**  提高 Context Utilization 的同时往往也会增加 Noise Sensitivity，这是一个需要权衡的问题。  可以通过调整 prompt、改进生成器模型等方式来寻求平衡。

* **关注不同生成器模型的特点：**  实验发现，不同的生成器模型对噪声的敏感性不同，例如 GPT-4 的 Noise Sensitivity 比开源模型更低。  因此，需要根据所使用的生成器模型的特点来调整策略。

总而言之，这些建议的核心在于**如何在信息量、噪声和生成器能力之间找到最佳平衡点**。  RAGCHECKER 提供的细粒度指标可以帮助开发者更好地理解这些因素之间的相互作用，并针对具体问题进行优化。
```
