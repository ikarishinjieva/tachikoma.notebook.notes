MySQL默认是开启死锁检测的，一旦发生死锁，InnoDB会回滚其中一个事务，将锁解放。默认show engine innodb status
会记录上一次的死锁日志，

也可以设置innodb_print_all_deadlocks 将每一次死锁的日志记录到error log中。

死锁日志中列出了死锁发生的时间，以及导致死锁的事务信息（只显示两个事务，如果由多个事务导致的死锁也只显示两个），并显示出每个事务正

在执行的 SQL 语句（事务执行多个SQL，只会记录正在执行的那个）、等待的锁以及持有的锁信息等。

 **死锁日志的局限性：**

  1. 不显示事务1已经持有了什么锁；
  2. 不显示事务所有的SQL，也就没法推测事务1已经持有了什么锁。

# MySQL处于等待中的连接

mysql> select
THREAD_ID,PROCESSLIST_ID,PROCESSLIST_USER,PROCESSLIST_HOST,PROCESSLIST_DB,PROCESSLIST_COMMAND,PROCESSLIST_TIME,PROCESSLIST_STATE,PROCESSLIST_INFO,THREAD_OS_ID  
from performance_schema.threads  
where PROCESSLIST_COMMAND='query' and PROCESSLIST_TIME !=0;

# 3、解决办法

被InnoDB行锁阻塞的情况一般比较少，因为默认锁等待时间很短（innodb_lock_wait_timeout），而且行锁影响范围小，一般观察不到。

出现的话可能也是因为人为执行了一个大事务，行锁较多，或忘记提交事务，导致锁一直未释放，此时也可以选择kill那个连接，解除阻塞；

考虑一个阻塞源可能阻塞多个线程，所以阻塞其他线程最多的那个连接就是阻塞源，通过下面的SQL进行查看：

    
    
    select sql_kill_blocking_connection,count(*) from sys.innodb_lock_waits group by sql_kill_blocking_connection order by count(*) desc;
    +------------------------------+----------+
    | sql_kill_blocking_connection | count(*) |
    +------------------------------+----------+
    | KILL 9                       |        3 |
    | KILL 12                      |        2 |
    | KILL 13                      |        1 |
    +------------------------------+----------+  
  
---  
  
此示例中 kill 9;即可，执行后查看阻塞是否解决，如果没有，重复以上步骤。

# 工单复盘整理：

# 1、问题总结

客户希望协助帮忙看下他们一套MySQL主从同步慢的原因。

# 2、现象描述

生产环境，MGR一主多从，开启了并行复制，有出现死锁和事务锁等待的现象，死锁问题MySQL会自动解决，事务锁等待的问题会导致后续事务无法回放，导致主从延迟。

# 3、解决方案

（1）有3个事务之间互相持有事务锁（11328101锁住了11328102，11328096锁住了11328101和11328099），由于事务涉及的具体SQL还不太清楚，建议开启慢查询进行观察。  
（2）存在索引重复的现象，MF_ID列已经作为主键，又在该字段创建了唯一索引，建议删除这个唯一索引（锁都是加在IDX_1这个唯一索引上了）后，观察锁争用的情况是否改善。

已将以上结论和建议同步给客户，客户反馈会让业务人员进行优化。

工单复盘整理

# 一、问题总结

  * 在mysql的主从架构中，某些正常的sql语句可能会出现偶发慢的情况，主库上的insert和update语句在高并发的情况下可能会导致锁等待时间增加，从而使sql的总耗时也增加。
  * 在从库上的select语句有时会执行较慢，可能是未命中buffer pool导致。

# 二、问题现象

如下sql的执行时间未达到预期，耗时较长。  
1.根据订单id查询订单(4次)

    
    
    SELECT XXX FROM t_order a WHERE a.id = '1610266843954055254'
    

2.根据订单id查询订单子项(10次)

    
    
    SELECT XXX from t_order_item where order_id = '1610278027015054987';
    
    

3.根据订单id查询订单状态(1次)

    
    
    select status from t_order where id = '1610267805954058527'
    
    

4.更新订单状态(8次)

    
    
    update t_order SET status = '212', update_time = '2021-01-10 07:11:57.518' WHERE id = '1610233531288047192';
    
    

5.插入订单支付表(3次)

    
    
    INSERT INTO t_order_pay_info(XXX) values (XXX)
    
    

# 三、问题原因

对反馈的语句进行了手工执行，速度均在0.00s级别以下；通过explain检查查询SQL的执行计划，使用的都是主键索引。  
尚未明确真实原因，以下为推测原因：

## 1，主库：update、insert语句

1，推测原因：innodb lock锁等待  
2，MySQL在对行进行修改时，需要先获取表的意向排他锁，以及此行的排它锁。在对表格的多行数据进行修改时，锁等待的时间将会增加。  
目前慢查中涉及到修改的表：t_order、t_order_pay_info均涉及到多次的更新。

## 2，从库：select语句

1，推测原因：innodb buffer pool中未命中查询数据  
2，考虑到目前慢查的时间分布并无明显的规律，推测部分超过200ms的耗时的查询语句，在执行时innodb buffer pool中没有相关的数据。  
MySQL在执行查询语句时，如果相关数据已经从innodb buffer
pool的内存页中刷入了磁盘，那么此时MySQL需要从磁盘中进行数据读取，再对数据进行发送。
